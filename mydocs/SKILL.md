# MyDocs 文档规范技能

## 版本

- 当前版本：`v1.0.0`
- 版本策略：语义化版本（`MAJOR.MINOR.PATCH`）

## 概述

`mydocs` 为本仓库定义统一的文档工作流。
目标是让 `docs/` 结构稳定、命名一致，并与代码变更保持同步。
默认产出的文档同时服务两类读者：开发人员和 AI（Claude Code、Codex）。
其中首要目标是让 AI 能准确理解、严格执行，并尽量一次性高保真落地实现。

## AI 高保真执行原则（强制）

文档必须具备“可直接执行”的信息密度，避免含糊描述。至少明确：

1. 输入与前置条件（依赖、环境、数据来源）
2. 输出与验收标准（可验证、可量化）
3. 约束与不变量（必须/禁止）
4. 主流程与异常恢复（失败后的可执行动作）
5. 数据契约与接口契约（字段、类型、边界）

写作要求：

- 优先使用“必须/禁止/仅当/否则”这类确定性措辞。
- 对关键步骤使用编号列表和检查清单，减少歧义。
- 避免“尽量/适当/可能”等无法执行的词，必要时给默认值。

## 工作模式

### 模式 A：新增（add）

- 用于新增功能域文档与首次初始化结构。

### 模式 B：更新（update，默认）

- 用于更新已有文档内容，不创建同主题重复文档。

### 模式 C：文档评审（review）

- 用于评审已有文档质量，不默认修改文件。
- 触发关键词示例：`review`、`评审`、`审查`、`架构评估`。
- 评审输出必须先给问题清单，再给结论。

## 执行策略：自动优先 + 风险闸门确认

默认策略是自动执行，不做多余确认，以最大化执行效率与一次性完成率。

仅在以下高风险场景触发用户确认后再执行：

1. 删除或重命名现有文档文件。
2. 将文档状态标记为 `superseded`（涉及主事实切换）。
3. 修改全局规范文件（如 `docs/DOCUMENTATION_STANDARDS.md`）。
4. 跨多个功能域批量改动（可能影响面较大）。
5. 需求语义不清或存在多种合理解释，继续自动执行可能误改。

未触发以上场景时，`add`、`update`、`review` 均应自动执行。

## 执行前置条件（强制）

执行任何文档动作前，必须先校验并读取以下文件：

- `docs/README.md`
- `docs/DOCUMENTATION_STANDARDS.md`
- `docs/DOC_VERSION_LOG.md`

若任一文件不存在，必须立即中止，并提示用户先创建后再继续。

缺失文件初始化模板路径：

- `.claude/skills/mydocs/resources/README.md`
- `.claude/skills/mydocs/resources/DOCUMENTATION_STANDARDS.md`
- `.claude/skills/mydocs/resources/DOC_VERSION_LOG.md`

## 规则优先级（支持覆盖）

规则按以下顺序生效：

1. `docs/DOCUMENTATION_STANDARDS.md`（最高优先级）
2. `docs/README.md` 中的本仓库补充约束
3. `mydocs` 技能中的基础版规则（兜底）

当 `docs/` 中有明确规则时，技能内同类规则自动被覆盖，不得反向覆盖 `docs/`。

## 必要目录结构

```text
docs/
├── README.md
├── DOCUMENTATION_STANDARDS.md
├── DOC_VERSION_LOG.md
├── _templates/
│   └── arch-spec-template.md
├── plans/
├── tech/
└── {feature-domain}/
    ├── PRD.md
    ├── arch-spec.md
    ├── DEV_TECH.md
    ├── DEV_PLAN.md
    └── CHANGELOG.md
```

## 基础版规则（仅兜底）

以下规则仅在 `docs/` 规范文件未明确覆盖时生效：

- 架构文档命名使用 `arch-spec`
- 阶段性文档命名使用 `YYYY-MM-DD-{topic}-{doc-type}.md`
- `arch-spec.md` 使用固定章节顺序
- 文档元信息包含“生成时间 + 版本”
- 仅关键性变更写入 `docs/DOC_VERSION_LOG.md`

## 资源模板（内置）

`mydocs` 内置三份基础模板，用于初始化 `docs/` 规范文件：

- `resources/README.md`
- `resources/DOCUMENTATION_STANDARDS.md`
- `resources/DOC_VERSION_LOG.md`

这些模板是默认基线。项目可在 `docs/` 中自由覆盖和细化，覆盖后以 `docs/` 为准。

## 命名规则

1. 长期维护文档使用 `docs/{feature-domain}/` 下固定文件名。
2. 阶段性一次性文档使用 `YYYY-MM-DD-{topic}-{doc-type}.md`。
3. 架构文档统一使用 `arch-spec`，不再使用 `design`。
4. 禁止模糊命名，如 `new.md`、`final.md`、`v2-final.md`。

## arch-spec.md 固定章节顺序

1. 背景、目标、范围、非目标
2. 整体架构与不变量
3. 核心流程与异常恢复
4. 数据模型与迁移
5. API 与事件契约
6. 组件与模块边界
7. UI/UX/UE 与信息架构
8. 可观测性与验收标准
9. 发布、回滚与分期计划
10. 风险与开放问题

## 元信息规则

所有新建或大改文档必须包含：

- 生成时间：`YYYY-MM-DD HH:mm:ss`
- 版本：`vX.Y`

## 最新性与版本日志规则（强制）

1. `docs/plans/` 允许保留历史阶段文档（时间序列归档）。
2. `docs/` 其他路径应保持“最新单一事实版本”，避免并存多个“final/v2/new”文档。
3. 关键性文档变更后，必须同步更新 `docs/DOC_VERSION_LOG.md`：
   - 记录文件路径、版本号、变更摘要、更新时间、状态（latest/superseded）。
4. 如果文档被新文档替代，旧文档必须在版本日志中标记为 `superseded` 并指向替代文件。

## 关键性变更判定（写入版本日志）

以下情况视为关键性变更，必须写入 `DOC_VERSION_LOG.md`：

1. 新增或废弃文档（状态变更为 `latest` / `superseded`）。
2. 架构/流程/数据模型/API 契约发生变化，可能影响实现或联调。
3. 验收标准、SLO、上线/回滚策略发生变化。
4. 文档主版本或次版本升级（如 `v1.2 -> v1.3`、`v1.x -> v2.0`）。

以下不要求写入版本日志（可仅改文档头部时间）：

- 错别字、排版、措辞润色
- 不改变语义的示例补充
- 小范围注释优化

## 更新策略（SSOT）

1. 同主题优先更新已有文档，不创建重复文档。
2. 新需求与旧文档冲突时，先更新文档，再改代码。
3. 功能实现完成后，文档需在同一变更中同步更新。

## 文档评审规则（新增）

### 评审维度

1. 结构完整性：是否符合 `arch-spec` 固定章节顺序。
2. AI 可执行性：步骤是否确定、输入输出是否清晰、是否可一次性执行。
3. 架构正确性：边界、约束、不变量是否清晰且可执行。
4. 流程闭环：主流程、异常流程、失败恢复、回滚是否完整。
5. 数据与契约：数据模型、迁移、API/事件定义是否一致。
6. 可运维性：日志、指标、告警、SLO、验收标准是否可落地。
7. 最新性与版本一致性：除 `docs/plans/` 外是否存在过期/重复文档，版本日志是否同步。

### 问题分级

- `P0`：阻断级，可能导致错误实现或上线事故，必须先修复。
- `P1`：高优先级，存在明显设计缺陷或遗漏，建议本轮修复。
- `P2`：优化项，不阻断交付，但建议纳入后续迭代。

### 评审输出格式（强制）

1. Findings（按 `P0` → `P1` → `P2`）
2. Open Questions（待确认假设）
3. Summary（是否建议通过）

每条 Finding 应包含：

- 严重级别
- 文档路径与定位（章节或行号）
- 问题描述
- 影响范围
- 修复建议

如果没有问题，必须明确写“未发现阻断问题”，并补充剩余风险。

## 执行清单

1. 识别 `feature-domain` 与目标文档。
2. 若 `docs/README.md`、`docs/DOCUMENTATION_STANDARDS.md` 或 `docs/DOC_VERSION_LOG.md` 缺失，先中止并提示按资源模板创建。
3. 识别当前是 `add`、`update` 还是 `review` 模式。
4. 按“自动优先 + 风险闸门确认”策略执行：非高风险动作自动完成，高风险动作先确认。
5. `add/update` 模式：应用命名规则与章节顺序规则；其中 `update` 优先修改已有文档，避免重复文档。
6. `add/update` 模式：必要时更新版本号、时间戳、索引；若触发关键性变更再更新版本日志。
7. `review` 模式：检查 AI 可执行性与最新性；仅输出评审结论，不默认改文档；如用户要求再给修订补丁。
8. 输出结果：变更清单（`add/update`）或评审清单（`review`），并明确版本日志是否“需要更新/已更新/无需更新”，以及是否触发风险确认。
